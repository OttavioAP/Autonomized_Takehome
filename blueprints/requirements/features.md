# Features

Feature tracker for the Team Activity Monitor, split along two axes — **functional vs. non-functional**, and **MVP vs. non-MVP** — with a status column per lifecycle stage. Order of implementation, priority, and tech stack are deliberately out of scope for this document.

**Status columns:** Specced / Implemented / Tested / Deployed / QA. Mark ✅ when done, ⬜ when not, 🟡 when partially done (see the row's own note for what's missing). See `CLAUDE.md` for the exact definition of each status and for when this file should be checked and updated.

## MVP

### Functional Requirements

| # | Feature | Description | Specced | Implemented | Tested | Deployed | QA |
|---|---------|-------------|:---:|:---:|:---:|:---:|:---:|
| MVP-FR-1 | Login screen (Azure SSO) | Authenticated entry point to the app; users sign in with their organization's Azure identity rather than a locally-managed credential store. Full OIDC confidential-client round-trip: `GET /auth/login` (redirect to Azure with state param + `prompt=select_account` so a browser with an existing Azure session doesn't silently skip account selection), `GET /auth/callback` (code exchange, JWKS signature validation, session row creation, cookie set), `login.html` template (dark console theme, `auth-shell`/`auth-card` layout). `get_current_user` dependency protects `/` and redirects unauthenticated HTML requests to `/login`. Automated tests pass (`tests/test_auth.py`, including a live-endpoint regression test against Microsoft's real authorize URL); manually verified live on the deployed site by the user against real Azure AD accounts (John/Sarah/Mike), including switching accounts via `select_account`. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-2 | Sign-out | Ends the user's session and invalidates any tokens/credentials derived from it, rather than only clearing client-side state (see MVP-NFR-5). Implemented as `POST /auth/logout`: CSRF-token-protected, marks the `sessions` row `revoked_at` and clears the cookie — `get_current_user` then rejects the revoked session on any subsequent request. Real bug found and fixed live: this only ever revoked the app's own session, never Azure AD's own browser SSO session, so signing out and back in as a different account silently re-authenticated as the same user — fixed via `prompt=select_account` on the login redirect (see MVP-FR-1). Automated tests pass; manually verified live by the user. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-3 | Chat window | The primary interface: free-text input and conversational output where users type natural-language questions and receive synthesized answers. Specced in full by `chat.md`. `app/templates/chat.html` is the real chat page — message log, input form, vanilla-JS SSE-over-POST reader driving `POST /conversations/{id}/chat` — wired to `GET /conversations/{id}`/`GET /`'s redirect. Full dark "console" theme UI pass: navbar (app name, Connections link, sign out), collapsible conversations sidebar, send-to-first-token throbber, and the chat pane fills the viewport via flex sizing (fixed a real overflow bug where the layout ran taller/wider than the browser at non-reference screen sizes). Verified end-to-end both via real HTTP exchange (httpx) and live in a real deployed browser by the user (multiple rounds of live testing and bug-fixing this session). | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-4 | Basic selectable components for JIRA/GitHub items | Chat responses render referenced items as clickable pills that deep-link to the source system — expanded well beyond the original 3 kinds during implementation to 9 total: `jira_ticket`, `jira_comment`, `jira_project`, `jira_person`, `github_commit`, `github_pr`, `github_comment`, `github_repo`, `github_user`. Project/person/repo/user pills were a real gap found via live user testing (pills worked for tickets/PRs/commits/comments but JIRA projects, GitHub repos, and people mentioned in an answer were plain unlinked text) — closed by making `discover_scope()` upsert each discovered project/person/repo/collaborator into `activity_items` with a real, live-verified deep-link URL (JIRA project boards, JIRA's people-directory page, GitHub repo/user pages), through the same citation trust boundary as every other kind. Pills also got a real visual pass (hover lift/shadow, underline, `max-width` + ellipsis so a long label can't overflow the bubble, `title` tooltip with the full text) and now render inside real markdown-parsed HTML (see MVP-FR-11) rather than plain text nodes. Richer inline summaries beyond a pill are still deferred to NMVP-FR-3. Verified live end-to-end repeatedly, including a real citation round-trip for each of the 9 kinds. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-5 | Natural-language query parsing | Extracts the target team member's name and the intent of the question (JIRA-only, GitHub-only, or combined) from a free-text query, so the right data-fetching path can be invoked. Deliberately not built as a parser — per `chat.md`'s design, the model reads the roster/discovered-people context directly and resolves who's being asked about itself; MVP-FR-11 covers this end-to-end. Handling multiple phrasings/formats robustly is deferred to NMVP-FR-6. | ✅ | ✅ | 🟡 | ✅ | 🟡 |
| MVP-FR-6 | Empty/error states in UI | Distinct, user-legible states for "member not found," "no recent activity," and "upstream API failure." Scoped down with the user to real upstream failures specifically (member-not-found/no-activity are already handled adequately in the model's own prose per `chat.md`'s design, no dedicated UI state needed). Real gap found and fixed: `UpstreamProviderError` was defined but never raised, and `ChatService`/`JiraTool`/`GithubTool` only caught `httpx.HTTPStatusError`, not connection-level failures (`httpx.ConnectError`/`TimeoutException`/DNS) — an unreachable OpenRouter/JIRA/GitHub would have crashed the SSE stream with an unhandled exception instead of degrading gracefully. Fixed by widening every relevant `except` to `httpx.HTTPError` (the common base) and giving the `error` SSE event a real bordered/iconed banner treatment (`.chat-error`) instead of bare red text; the user-facing message is now a clean, actionable summary with the full exception logged server-side, not a raw exception repr. New live-forced tests (no mocks — real unreachable-host connections against `127.0.0.1:1`) for all three upstream legs. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-7 | Conversational query history within a single conversation | The chat retains context across turns within one conversation (e.g., follow-up questions about the same person), but does not persist across sessions — that's NMVP-FR-1. `messages`/`message_citations` persist every turn, `message_repo.list_out_for_conversation` replays full history with citations pre-joined on `GET /conversations/{id}`. History replay now renders markdown identically to the live view (see MVP-FR-11). **Bug fixed this session:** prior turns were persisted and replayed *in the UI*, but were never fed back to the *model* — `ChatService.run()` built its LLM message list from only the system prompt + current query, so follow-ups had no memory of earlier turns. Now `run()` loads `message_repo.list_for_conversation` and splices prior turns (with `{{cite:...}}` sentinels stripped via `_strip_cite_sentinels`) between the system prompt and the current user turn. Covered by `test_run_replays_prior_conversation_history` and `test_strip_cite_sentinels_*`. Verified live via httpx and real browser use. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-8 | Static mapping of team members to JIRA/GitHub identities | A seed dataset resolving each team member's display name to their JIRA account and GitHub handle, since Azure SSO identity, JIRA identity, and GitHub identity are three separate namespaces. Real design correction found via live use: the seeded `jira_account_email`/`github_login` are only a *starting default*, not an enforced identity — a team member can legitimately connect any JIRA/GitHub account (per `oauth-integration.md`'s actual design). An earlier fix in this session incorrectly rejected a GitHub connect whose account didn't match the seed value; corrected to instead update `team_members.github_login`/new `jira_account_id` column to whichever account actually authorizes, so pre-fetch/the chat roster always look up the real connected identity, not a stale seed. `team_members` is a real Postgres table (not just the JSON fixture), seeded via `scripts/seed.py` on every boot. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-9 | Fetch JIRA data for a user | Retrieves a user's assigned issues plus issue status, priority, type, recent updates, and comments from the JIRA REST API, using either the static identity mapping (MVP-FR-8), the connect-time-resolved `account_id`, or a discovered project member. Specced in `chat.md`'s Tools section (`JiraTool`) and `oauth-integration.md`'s Client rework/Scope discovery sections. `app/integrations/jira_client.py` has `search_projects`/`search_assignable_users`/`get_comments`/`get_current_user_account_id`, a `since_days` bound on `get_issues_assigned_to`, and real connection-failure handling (`httpx.HTTPError`, not just `HTTPStatusError`). Live-verified against the real `autonomizedtest1.atlassian.net` instance across 3 real JIRA projects (KAN, MOB, DATA) seeded with 23+ issues and comment threads. `JiraTool.execute()`'s Bearer/`cloud_id` path is now genuinely live-verified — real JIRA OAuth connections exist on the deployed site (John/Sarah/Mike), not just Basic-auth-equivalent unit coverage. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-10 | Fetch GitHub data for a user | Retrieves a user's recent commits (date-bounded via `since`/`until`), active pull requests (with review decision), repos, contributors, and PR comments from the GitHub REST API. Specced in `chat.md`'s Tools section (`GithubTool`). `app/integrations/github_client.py` has `get_user_repos`/`get_repo_contributors`/`get_pr_reviews`/`get_issue_comments`/`get_authenticated_login`, plus real connection-failure handling. `GithubTool.execute()` is fully live-verified end to end across 3 real repos (`Autonomized_Test_Project_1`, `mobile-companion-app`, `analytics-pipeline`) with commits, PRs (open/merged, approved/changes-requested), and comments — client functions, the tool layer, and the full `ChatService`/route stack, including a real production concurrency bug found and fixed (`activity_item_repo.upsert`'s check-then-insert had a TOCTOU race once same-round tool calls started running concurrently — fixed with a real atomic `INSERT ... ON CONFLICT DO UPDATE`). | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-11 | Response synthesis | Combines the fetched JIRA and GitHub data into a single, coherent, human-readable chat answer — the core deliverable the whole system exists to produce. Implemented via `ChatService.run()`'s agentic loop (`app/services/chat_service.py`) — system prompt + tool-calling (same-round calls now run concurrently via `asyncio.gather`, each on its own DB session) + citation-sentinel streaming. Real bugs found via live use and fixed: (1) the model answered from thin pre-fetched context instead of calling a tool, and narrated its own internal process ("based on the pre-fetched data...") — fixed with a hard "always call a tool first" system-prompt rule and removing every "pre-fetched"/tool-referencing phrase from the prompt's own vocabulary; (2) no separator between a tool-call round's leading text and the next round's answer, producing run-together sentences — fixed with an explicit `"\n\n"` token between rounds; (3) the model's markdown output (`**bold**`, `- lists`) was never rendered, showing as literal syntax — fixed with `marked.js` (live) and Python-Markdown (`resolve_citations`, history replay), both escaping model text before parsing to prevent HTML injection. Live-verified extensively against real OpenRouter + real JIRA/GitHub data, both providers now fully exercised (no more JIRA-blocked gap). The system prompt now also instructs the model to surface tool failures to the user (JIRA/GitHub not connected, or an upstream fetch error) rather than silently dropping a platform or pretending the person has no activity — distinguishing an `Error:` tool result from a successful-but-empty one. | ✅ | ✅ | ✅ | ✅ | ✅ |
| MVP-FR-12 | Token streaming for chat responses | The chat displays partial response content as it is generated rather than waiting for the full answer. Streaming transport: SSE (`text/event-stream`, `event: <name>\ndata: <json>\n\n` frames), consumed client-side via a vanilla `fetch` + `ReadableStream` reader (not `htmx-ext-sse`, which is GET-oriented — this endpoint is a CSRF-protected POST). A visible throbber now covers the gap between send and first token/tool-status/cite/error event. Implemented and live-verified: `POST /conversations/{id}/chat` streams `token`/`tool-status`/`cite`/`cite-error`/`error` events progressively; time-to-first-token not formally tracked as a metric (out of scope for this pass). | ✅ | ✅ | ✅ | ✅ | ✅ |
