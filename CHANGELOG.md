## 2026-07-20 (Fix: production `team_members` never seeded, blocking every real login)

- Real production incident, caught by the user testing the live deployed site: Azure AD SSO login succeeded but every page returned `{"detail":"No team member record for this account"}"`. Root cause: `deploy.yml` only ever seeded a throwaway CI Postgres container (to satisfy `tests/test_team_members_seed.py`), never the real Azure Postgres Flexible Server behind prod — same class of gap as this project's earlier documented "migrations never ran against prod" bug, and the same fix applies: `python scripts/seed.py` now runs in `entrypoint.sh` on every container boot (right after `alembic upgrade head`), since Postgres Flexible Server's firewall only allows Azure-internal traffic, no path for a GitHub-hosted CI runner to reach it directly.
- `scripts/seed.py` made idempotent (`INSERT ... ON CONFLICT DO NOTHING` on each seeded table's natural key) before wiring it into unconditional every-boot execution — it previously used a plain `INSERT`, which would have thrown `UniqueViolation` and crashed the container on every restart after the first successful seed.

## 2026-07-20 (Phase 4: OpenRouter tool-calling)

- Extended `app/services/llm_router.py` with streaming tool-call support, per the revised `openrouter-integration.md` (`chat.md`'s upcoming `ChatService` agentic loop depends on this). New Pydantic types: `ToolDefinition` (callers pass a Pydantic model class for `parameters`, not a hand-written JSON schema), `ToolCall` (raw JSON `arguments` string plus `parsed_arguments(as_type)` for typed validation at the boundary), `TextDelta`/`ToolCallDelta`/`StreamDone`/`QueryEvent` (the new discriminated-union return type of `query()`, replacing bare `str` chunks). `ChatMessage` gained a 4th `"tool"` role plus `tool_calls`/`tool_call_id` fields, `content` now `str | None`.

Streaming tool-call deltas are fragmented across SSE chunks (id/name on the first fragment per `index`, argument pieces on every fragment at that index) — accumulated into a local buffer and yielded as one complete `ToolCallDelta` only once the terminal `finish_reason` chunk arrives, followed by `StreamDone`.
- Real bug caught by the new live test (not by inspection): OpenRouter sends at least one more chunk after the one carrying `finish_reason` for a tool-call turn, still carrying the same non-null `finish_reason` — without an explicit `return` after the first terminal chunk, the accumulated `ToolCallDelta` was yielded twice. Fixed by returning immediately after the first terminal `finish_reason` is observed.
- `tests/integrations/test_llm_router.py`: updated the 2 existing live tests for the new event-based return type (filter for `TextDelta`, assert trailing `StreamDone(finish_reason="stop")`), added a 3rd live test exercising a real tool-call round trip (trivial `city: str` params model, prompt asking to call a weather tool for Boston) — asserts one `ToolCallDelta`/one `ToolCall`, `StreamDone(finish_reason="tool_calls")`, and successful `parsed_arguments(...)` round-trip. All 3 pass against live OpenRouter.

## 2026-07-19 (post-deploy: prod login 500)

- Fixed a real bug found via manual browser testing on the deployed instance: clicking "Sign in with Microsoft" completed the full Azure login round-trip successfully, but `/auth/callback` 500'd writing the session row — `asyncpg.exceptions.UndefinedTableError: relation "sessions" does not exist`. Root cause: `deploy.yml` only ever ran `alembic upgrade head` inside the throwaway `test` job's CI Postgres container, never against the real Azure Postgres Flexible Server the deployed app points at — so prod's schema was still at the original `smoke_test`-table migration from the very first deploy, predating every feature table added since (including `sessions`). Diagnosed by enabling `az webapp log config --application-logging filesystem` (was off) and tailing live logs (`az webapp log tail`) while reproducing the callback 500 directly via curl with a real session cookie.
- Considered running the migration as a discrete `deploy.yml` step (either via a GHA-hosted runner reaching Postgres directly, or via Kudu's in-Azure `/api/command` to dodge the Flexible Server firewall, which only allows Azure-internal traffic and would reject a GitHub-hosted runner). Went with a different, more standard fix instead: **migrations now run as part of every container boot**, both local and deployed. Added `entrypoint.sh` (`alembic upgrade head && exec uvicorn ...`) and changed the `Dockerfile`'s `CMD` to `["./entrypoint.sh"]`. Single-instance deployment (see `blueprints/deployment.md`), so no leader-election/lock concern from multiple replicas racing to migrate concurrently. `alembic upgrade head` is a no-op when already current, so this adds negligible boot time and makes `make migrate`/CI's explicit migrate step redundant-but-harmless rather than something to remove.
- The next push/redeploy picks up the new entrypoint automatically and self-heals prod's schema on boot — no manual one-off migration command needed against the live database.
