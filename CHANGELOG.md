## 2026-07-19 (earlier)

- Implemented timeline.md step 3 (stand up minimum structure/infra) per `blueprints/specs/stack-and-infra.md`: layered `app/` package (api/services/repositories/integrations/models/db), SQLAlchemy async `Database` singleton with connection pooling, hello-world htmx+Jinja2+Pico page proving the full fragment-swap request path, Dockerfile (non-root, host-UID-matched via build args, `pip-freeze` lockfile, editable install for live reload) and docker-compose, and a Makefile wrapping it all. Recorded the routes-own-the-DB-transaction invariant in `CLAUDE.md` (not `stack-and-infra.md`) since that file is injected into every agent's context automatically.
- Alembic wired up with the async template (`connection.run_sync` bridging), `target_metadata` sourced from `app.db.models`, DB URL pulled from `app.config` rather than duplicated in `alembic.ini`, and a `process_revision_directives` hook that skips writing a migration file entirely when autogenerate finds no real diff. A throwaway `SmokeTest` model proves the pipeline end-to-end ahead of any real feature table. Makefile got a new `revision` target and `up-dev` now runs `migrate → revision → migrate` so schema changes since the last migration get captured automatically.
- `.pre-commit-config.yaml` existed but had never actually been installed as a git hook or run once — set up a persistent host `.venv` (git hooks run on the host, not in Docker), ran `pre-commit install` for real, and the first `pre-commit run --all-files` surfaced a genuine bug: the `-jinja` djLint hooks filter by `types_or: [jinja]` upstream (a `.jinja` file extension), so the `files: \.html$` override never matched anything since it's ANDed with, not a replacement for, that type filter. Fixed by overriding `types_or: [html]` directly; the resulting real findings (hardcoded static paths instead of `url_for()`, unnamed endblocks) were fixed on their merits rather than suppressed.
- `scripts/seed.py` derives load order from `Base.metadata.sorted_tables` (FK-safe, no hand-maintained table list) and no-ops cleanly with "nothing to seed" while `local-dev-data/` is empty, pending real JIRA/GitHub demo identities.
- pytest scaffolded with an async `httpx`/`ASGITransport` client fixture (matches the async-first stack, no `TestClient`). Verified to run not just via `make test` but standalone inside the built image (`docker run --rm <image> pytest`, zero compose/bind-mount/env) — proving the test suite travels with whatever gets deployed, not just local dev.
- Moved the `.github/workflows/deploy.yml` stub out of step 3's deliverable list and into step 4's — it's a deploy mechanism, not infra scaffolding, and step 4 already owns the rest of the Azure deploy story via `blueprints/deployment.md`.
- Deployed hello-world to Azure (timeline.md step 4, MVP-NFR-1), verified live at `https://team-activity-monitor-a8b9a7.azurewebsites.net/` (both `/` and the htmx `/ping` fragment route return correct responses). Wrote `blueprints/deployment.md` capturing the agreed plan up front, plus `scripts/azure/{lib,provision,deploy,teardown}.sh` — idempotent, committed, reusable scripts rather than ad hoc commands.
- Azure CLI installed into an isolated `.venv-azure/` (pip, no sudo/system package) so it doesn't pollute the app's own dependency chain.
- Discovered and worked around two distinct new-subscription quota restrictions: Postgres Flexible Server blocked in `eastus` under Free Trial quota (fixed by upgrading to Pay-As-You-Go billing), and a separate 0-VM App Service compute quota in `eastus`/`eastus2`/`westus` unaffected by that upgrade (fixed by pinning compute resources — Postgres, App Service plan/webapp — to `centralus`, while the resource group and ACR stay in `eastus`). Both are recorded in `blueprints/deployment.md`'s execution notes so a future re-provision doesn't rediscover them the hard way.
- Fixed a `deploy.sh` bug where `.env` was `source`d directly, causing real secret values containing shell-special characters (`$`, `#`) to break under `set -u`; now parsed line-by-line as inert text.
- No domain purchased — using Azure App Service's free `*.azurewebsites.net` hostname, appropriate for a throwaway ~1-week deployment. Postgres network access restricted to "allow Azure services" rather than the full public internet.
- Fixed a mixed-content bug on the deployed site: Azure terminates TLS at its edge and forwards to the container over plain HTTP, so `url_for()` generated `http://` asset links on the `https://` page, which browsers blocked. Fixed via `--proxy-headers --forwarded-allow-ips=*` on the Dockerfile's `uvicorn` CMD so it honors Azure's `X-Forwarded-Proto` header. Diagnosed with a temporary `/debug-headers` route (added, used, then removed).
- Found and worked around an intermittent issue where `az webapp config container set` + restart leaves the site reporting `Running` but actually unresponsive (Docker log shows an interrupted image pull); `deploy.sh` now issues a second restart automatically, which has reliably recovered it both times it's been hit.
- At the user's request, revised the CI/CD decision from `stack-and-infra.md` (originally: no auto-deploy, manual `workflow_dispatch` only, no standing Azure credentials in GitHub): `.github/workflows/deploy.yml` now auto-deploys on every push to `main`, gated on a `pytest` job passing first. Created a resource-group-scoped Azure service principal and wired it plus `DATABASE_URL`/`OPENROUTER_API_KEY` into GitHub Actions secrets. Full rationale and secret inventory in `blueprints/deployment.md`.
- First CI run took 4 attempts to go green, two real bugs found and fixed: (1) `ACR_NAME`/`ACR_LOGIN_SERVER`/`AZURE_RESOURCE_GROUP`/`AZURE_WEBAPP_NAME` secrets had been set with `echo` instead of `printf`, baking in a trailing newline that broke `az acr login` with a misleading "resource not found" error; (2) ad hoc local testing of the service principal's credentials accidentally deleted the one `AZURE_CREDENTIALS` actually referenced. Both fixed and documented in `blueprints/deployment.md`'s execution notes. CI/CD auto-deploy now verified working end-to-end: push → test job → deploy job → live site confirmed at 200 OK on `/` and `/ping`.
