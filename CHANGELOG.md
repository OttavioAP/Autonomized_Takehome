Once `get_current_user` made that import unavoidable, CI started failing on missing required Settings fields. Added a `postgres:16` GHA service container, `alembic upgrade head` before `pytest` inside the built image (via `--network=host` to reach the service), and wired in 10 new GitHub Actions secrets (`JIRA_BASE_URL`, `JIRA_PROJECT_KEY`, `JIRA_EMAIL`, `JIRA_API_TOKEN`, `GH_INTEGRATION_TOKEN`/`GH_INTEGRATION_REPO` ŌĆö `GITHUB_`-prefixed secret names are reserved by GitHub, `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET`, `AZURE_REDIRECT_URI`/`AZURE_REDIRECT_URI_PROD`) that the `deploy` job's appsettings step was already referencing before they existed (a pre-existing gap predating this session, surfaced while fixing the immediate regression).
- Regenerated `requirements.lock.txt` from the venv (`pip freeze`) after adding `python-jose` ŌĆö the Docker build installs from this lockfile, not live from `pyproject.toml`, so a stale lockfile silently meant the new dependency was never actually present in the container despite being declared.

- Removed the throwaway `SmokeTest` model/table now that a real model (`sessions`) exists to prove the Alembic pipeline; the migration file itself is left in history (migrations are append-only) and the new autogenerated migration drops the table.
- Extended `scripts/azure/teardown.sh` to also delete the 3 Azure AD users and the app registration (tenant-level objects, not resource-group-scoped, so `az group delete` never touched them) ŌĆö pulled from `.env` rather than `scripts/azure/.state`, since that file only tracks resource-group-scoped resource names.
